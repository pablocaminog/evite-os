Architecture Approaches

Pick the structure for evite-os

A

Full SSR Astro

One Astro app, @astrojs/cloudflare adapter, SSR mode. API routes = Astro endpoints. D1/R2/Email/Telnyx accessed server-side. Single Worker + Static Assets deployment.

Pros

  • One repo, one deploy
  • No CORS, no API glue
  • Simplest mental model

Cons

  • All pages dynamic (slower cold start)
  • Tighter Astro/API coupling
B

Hybrid Astro (Recommended)

Astro in hybrid mode — static pages pre-rendered (landing, docs), SSR pages dynamic (dashboard, RSVP, invitation). API routes as Astro endpoints. Best performance + flexibility.

Pros

  • Landing page instant (static)
  • Dynamic where needed
  • Still one deploy
  • Scales well as features grow

Cons

  • Need to tag pages export const prerender
  • Slightly more config
C

Static Astro + Separate API Worker

Astro builds pure static HTML/JS. Separate Hono or plain Worker handles all /api/*. Two Workers or one with routing. Full separation of frontend and backend.

Pros

  • Pure separation of concerns
  • API fully independent

Cons

  • Two deploys to manage
  • CORS needed
  • RSVP pages need client-side fetch
  • Most complexity for little gain

⭐ Recommendation: B — Hybrid Astro — static landing + SSR where it matters, one deployment.
